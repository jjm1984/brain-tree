# How to Run a Brain Tree Session

This is a skill for Claude, not for J — the process to follow when
acting as thinking partner and filing agent for the tree, whether
mining backlog, placing a new idea, or building on existing content.
Read alongside the project instructions and `meta/principles.md`.

## Before adding anything

1. **Search first, always.** `project_knowledge_search` before web
   search, before assuming, before extending.
2. **Read the actual node before extending it.** Not a summary, not a
   memory of it, the file itself. See `meta/staging-protocol.md`
   "Read-before-extending rule" — this is not optional, it caused a
   real error in this project when skipped.
3. **When a request is ambiguous about which existing node it touches,
   ask before building extensively** — a short, mobile-friendly
   question, not a wall of clarifying questions. Two live examples
   from this project: an RSD backlog item initially misplaced as J's
   own pattern before correction; "Moore's staircase" initially
   misread as irreducibility before J clarified it meant accessibility
   hierarchy. Both were caught by asking rather than assuming, but
   both required a correction that could have been avoided by asking
   one step earlier.

## While building

4. **New claims travel with their structural apparatus, not after.**
   A claim without its counter, crossroads, or exposed-root tag isn't
   half-done — it's a preference presented as more settled than it is.
   Don't stage the claim and plan to add the counter later.
5. **Convergent evidence across independent, unrelated fields is a
   real upgrade**, and worth naming explicitly when it happens — from
   lived-experience-only toward grounded-standalone. It is not the
   same as citing more sources from the same tradition. Two fields
   independently landing on the same finding (e.g. crisis
   de-escalation training and equine/canine calming signals both
   converging on the same body-language pattern, entirely separately)
   is strong; five listicles from the same genre of source is not,
   regardless of count.
6. **Test claims before staging them, not after.** When a claim is
   checkable (replication status, a named critique, a contested
   extension), check it before it goes in the tree, not as a later
   correction. If it fails the check, record the rejection explicitly
   (see staging-protocol) rather than quietly dropping it.
7. **Don't let a source's citation count buy it status.** Per
   `meta/principles.md` "Alignment-Frequency Is Not Engagement-Depth"
   — apply this actively, not just when it's already flagged. If a
   source is about to get cited a third or fourth time, check whether
   this citation is being evaluated on its own merits or fast-tracked
   because the source already has standing.
8. **Separate what a source actually established from what it's being
   used to argue.** When a source runs science into philosophy or myth
   without marking the seam (the standing critique already applied to
   Peterson in this tree), don't import the whole package — take the
   established part, tag it correctly, and leave the contested
   extension either out or explicitly flagged as contested.
9. **Test assumptions and play devil's advocate before agreeing** —
   standing user preference, not just a style note. This includes
   pressure-testing your own proposals, not only the person's.

## Structuring the work

10. **Multi-file sessions get a manifest.** See
    `meta/staging-protocol.md`. Don't hand-write dependency order in a
    closing message when a manifest can carry it durably.
11. **When a session runs long and touches many threads, do a closing
    audit before ending** — what was agreed but not staged, what was
    left open by design, what was offered and quietly dropped, what's
    pending manual action, what's untouched from before this session
    started. This project's practice: a dedicated audit pass at
    natural stopping points, not just at the very end.
12. **A prompt handed to Claude Code should state the guardrails, not
    just the task list** — what not to resolve, what counters must
    stay attached, what was explicitly rejected and shouldn't be
    re-added, what to report back rather than silently commit.

## On skills specifically

13. See `SKILL-CREATION.md` for the skill-authoring spec. Building a
    skill is itself a tree-building action and follows all of the
    above — a skill's claims need counters too, a skill built from
    convergent lived-experience-plus-external-sourcing is stronger
    than one built from either alone, and skill schema changes get
    proposed and reported, not silently applied.

## What this document is not

Not a replacement for `meta/principles.md` (crossroads, counters,
exposed roots, dear reader, tension holding, further reading) — those
are the tree's structural commitments. This document is the
*procedural* layer underneath them: how a session should actually run
so those commitments get honored consistently, session to session,
regardless of which chat instance is running it.
