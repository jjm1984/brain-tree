# The Life-Domain Audit

A periodic cross-branch life review, run in third-party/external-observer voice rather than first-person, as a deliberate friction point against a confirmed self-examination-avoidance pattern (see `leaf-just-cause-motivation-architecture`'s open-thread crossroads: "reading one's own motivation is harder than reading others'" — and `leaf-ability-translator-role`'s tied-together open question). Stage 1 of `map-life-direction-cycle` (Audit → Design → Strategy → Execution → Weekly Review).

Sourced from `backlog/life-audit.md`'s generic 8-area Instagram framework — the specific area list was discarded as content (unattributed, low-tier per this project's source-verification standard). What survived, and what this node is actually built from, is the third-party-voice mechanism plus a full restructure (2026-07-19) that absorbs `audit-framework`'s diagnostic machinery into every domain pass.

## Restructure summary (2026-07-19)

This node is now the parent. It absorbs `audit-framework`'s actual mechanism (four conditions of conscious sacrifice, measurable tripwires, written code, calibrated-person checkups, body-as-oracle, double-trigger architecture) as the shared WORTH-MACHINE TEST run inside every domain child below — not as a sibling domain alongside them. Third-party/external-observer voice governs the whole periodic pass. `audit-framework` itself remains the node of record for the diagnostic detail (see that file for full mechanism text) — this restructure changes its relationship to the tree (absorbed-into, not merely cross-connected-to), not its content.

## Cross-cutting checks

Three distinct check types, run across every domain child above — not domains themselves.

### 1. Worth-machine test (mandatory)

The same question inside every domain child. Absorbed directly from `audit-framework`: the four conditions (named value, proportionate cost, preserves capacity, articulable), measurable tripwires, double-trigger architecture (tripwire fires AND code says "do more" → the conflict itself is the signal). Third-party voice throughout. This is a TEST, not a domain — explicitly not a ninth sibling child.

### 2. Legend/legacy-worth weighting (always asked, answer can be null)

OPTIONAL in the sense of "check whether it's currently active here," NOT optional in the sense of "sometimes skip asking" — always asked, answer can legitimately be "not active in this domain," consistent with `leaf-legend-legacy-worth`'s existing note that it is domain-specific, not a constant background hum (confirmed active in vocation/army-record material, confirmed NOT active in the financial-reserve domain).

**Forcing point:** per this tree's own deferred-truth principle, an optional check with no forcing point risks silently never being asked anywhere. The forcing point IS this weighting step itself being a mandatory question (even though the answer can be "inactive") — the audit pass cannot skip past a domain without recording an explicit legend-worth answer, even a null one.

**Absorbed mechanism:** the "daughters as mirror" mechanism from `audit-framework` ("would I let her live the way I'm living?") is folded in here as one concrete instantiation of this check type, most naturally weighted in the parent-child domain but available as a technique across any domain.

### 3. Learning-growth resource check

NOT a domain child (explicitly decided against — this runs under everything rather than being folded into career specifically). A resource check, structurally distinct from the other two types: is this domain building or draining capacity to learn and adapt — not "is this honest" (worth-machine) or "does this carry legacy-weight" (legend-worth).

**Recursive note:** this tree's own construction is itself a learning-intellectual-growth activity — this audit-framework restructure, this conversation, is a live instance of the thing being audited.

**`skill_status` mechanism** — applies to `skill-*` nodes generally, and specifically informs Strategy/Execution requirements in `map-life-direction-cycle`. Field: `skill_status: integrated | outsourced-delegated | gap`. See `meta/SCHEMA.md` and `skills/SKILL-CREATION.md` for the field itself.

- **integrated** — already compiled into hexis, habituated, running below conscious effort (Aristotle's hexis/ethismos — named in `feedback/feedback-loop.md`, not yet planted into `philo-courage`, so cited here as a live reference, not a grounded root).
- **outsourced-delegated** — importable via hiring, delegating, a connector, a tool, another person's judgment. MUST route through worth-machine condition (c) (preserves vs. erodes capacity) in BOTH directions, not just one: (i) AVOIDANCE-HIJACK — "I'll just outsource this" as the worth-machine avoiding the discomfort of a genuine skill-gap, dressed as delegation; (ii) PRIDE-HIJACK — refusing to delegate, driven by pride/wanting-credit/legend-worth-adjacent need to be the one who did it, dressed as diligence or quality standards. Same decision point, worth-machine can hijack either the yes or the no — structurally identical to the double-trigger architecture's "conflict between two signals is itself the signal" pattern. Real convergent tradition, cited directly: Ricardo's comparative advantage — even if you're better at literally everything (absolute advantage), you should still delegate where your opportunity cost is highest, because the binding constraint is time, not skill.
- **gap** — a genuine capability that needs to be built or learned before the strategy/execution can proceed at all.
