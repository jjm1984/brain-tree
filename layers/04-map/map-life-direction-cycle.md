# Life Direction Cycle

A circular, generative cycle, not a linear one-time plan. Sourced from `forge/life-planning.md`'s generic 6-step framework — the raw artifact (unattributed "The Organized Money" Instagram content, same treatment as `map-life-domain-audit`'s source graphic) was discarded; what's kept is the mechanism, rebuilt against this tree's own existing content rather than the source's generic category list.

This resolves an earlier open placement question: the framework does NOT belong in `branch-daily-architecture` — that branch is single-day operational structure (circadian, hour-level anchors), a different granularity from life-direction/multi-month goal-setting. It also isn't a disconnected sibling to `map-life-domain-audit` — the two unify into one circular Map-layer cycle.

## The five stages

**1. AUDIT** — periodic honest cross-domain assessment, run in third-party/external-observer voice. See `map-life-domain-audit` for the full mechanism.

**2. DESIGN** — where is life aimed. Anchored to the trunk (do what is fulfilling, cause no harm), not to externally-borrowed ambition — this stage is the most vulnerable point for the worth-machine pattern (self-worth entangled with output) to quietly substitute an inherited goal for a chosen one. Design should be interrogated against that risk each cycle, not assumed clean.

**3. STRATEGY** — break Design into domains (health, finances, work, relationships, etc. — the same domain set as `map-life-domain-audit`, kept consistent across the cycle rather than re-derived each time), set goals per domain, build a planning system per domain.

**Audit-framework as a Design/Strategy criteria source (2026-07-19):** there's already a precedent in the tree for this — `branch-daily-architecture` is already explicitly connected to `audit-framework` ("the code provides the criteria for daily architecture... audit-framework: tripwires that require structural conditions to fire"). This cycle's Design and Strategy stages should draw on the same relationship — `audit-framework`'s four conditions are a plausible criteria-source for auditing Design-stage choices (is this vision genuinely fulfilling-and-harmless, or worth-machine drift in nicer language), the same way they already ground daily-architecture.

**4. EXECUTION** — tactical action. This is the Map-layer stage that Skill nodes (`skill-*`) decompose from, per the 2026-07-18 Leaf→Fruit→Map→Skill resolution (`meta/principles.md`). Skills should set their `parent_map` to this stage specifically, not to Design or Strategy — Execution is where a strategy becomes a tactic.

**Inherited tension (2026-07-19):** `philo-1pct-compounding` already contains the exact tension this stage needs to inherit, not re-derive: "the compounding doesn't start until the structure supports it" (stated there regarding the job-change/daily-architecture design window). The same structural precondition — Map/structure must exist before the tactical layer can compound — applies here.

**5. WEEKLY REVIEW** — a tighter-cadence check than the full periodic Audit. Functions as the forcing mechanism this tree's own deferred-truth principle requires: without a scheduled forcing point, "later" silently becomes "never." This is the mechanical answer to that named structural risk, applied at the goal-tracking level.

**Loop.** Weekly Review feeds back into the next periodic Audit, closing the cycle. Grace & Flexibility is not a separate stage but the property that lets Design and Strategy actually revise between cycles rather than calcifying — this is the trunk's belief-release facet (`trunk-belief-release-when-no-longer-serves`, "hold belief firmly, release it when it no longer serves you") operating specifically at the life-direction level, distinct from its epistemic application elsewhere in the tree.

## Convergence note

Structurally close to Boyd's OODA loop (Observe-Orient-Decide-Act, military strategy) and Deming's PDCA (Plan-Do-Check-Act, quality management) — two independent traditions converging on a similar circular structure. Worth naming per this tree's convergent-evidence-upgrade rule, though this cycle was arrived at independently in conversation rather than deliberately drawing on either tradition — a convergence observation, not a borrowed framework. Do not retroactively imply OODA/PDCA were sources. See `philo-ooda-decision-cycle` for the full OODA material, added the same day this convergence was first named.
